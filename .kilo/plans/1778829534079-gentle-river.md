# Plan: API Docs with Swagger OpenAPI + Scalar UI

## Goal
Add API documentation for the existing PizzaHub API using `@nestjs/swagger` to generate the OpenAPI document, but do not enable Swagger UI. Serve the documentation UI with Scalar instead, including auth endpoints and bearer token support so users can login/register and use the JWT token for protected endpoints.

## Proposed Routes
- `GET /docs` -> Scalar API reference UI
- `GET /openapi.json` -> raw OpenAPI JSON document
- Do not add `SwaggerModule.setup(...)`, so Swagger UI remains disabled.

## Dependencies
Install Scalar NestJS integration:
```bash
npm install @scalar/nestjs-api-reference
```
`@nestjs/swagger` is already installed.

## Implementation Steps

### 1. Add Swagger/Scalar setup helper
Create a reusable docs setup file, likely:
```txt
src/common/docs/swagger.config.ts
```
or:
```txt
src/config/swagger.config.ts
```

Recommended implementation responsibilities:
- Build `DocumentBuilder` config:
  - title: `PizzaHub API`
  - description: API documentation for auth, users, categories, and products
  - version: `1.0`
  - tags: `auth`, `users`, `categories`, `products`
  - bearer auth scheme using `.addBearerAuth(...)`
- Generate OpenAPI document via `SwaggerModule.createDocument(app, config)`
- Serve raw JSON at `/openapi.json`
- Serve Scalar UI at `/docs` using `apiReference({ url: '/openapi.json', theme: 'purple' })`
- Do not call `SwaggerModule.setup()`.

### 2. Wire docs setup in `src/main.ts`
Update `main.ts` after app creation and global validation pipe setup:
- import docs setup helper
- call `setupApiDocs(app)` before `app.listen(...)`

Keep existing `ValidationPipe` unchanged.

### 3. Add Swagger decorators to Auth DTOs
Update:
```txt
src/modules/auth/dto/register.dto.ts
src/modules/auth/dto/login.dto.ts
```

Add `@ApiProperty` / `@ApiPropertyOptional` for:
- `name`
- `email`
- `phone`
- `password`

Use examples:
- `name: Alfredo`
- `email: alfredo@example.com`
- `phone: 08123456789`
- `password: password123`

### 4. Add Swagger decorators to controllers
Update active controllers:
```txt
src/modules/auth/auth.controller.ts
src/modules/users/users.controller.ts
src/modules/categories/categories.controller.ts
src/modules/products/products.controller.ts
```

Add:
- `@ApiTags(...)` on controller classes
- `@ApiOperation(...)` on endpoint methods
- `@ApiBearerAuth()` on protected endpoints:
  - `GET /auth/me`
  - `GET /users/me`
- Optional response decorators:
  - `@ApiOkResponse(...)`
  - `@ApiCreatedResponse(...)`
  - `@ApiBadRequestResponse(...)`
  - `@ApiUnauthorizedResponse(...)`
  - `@ApiNotFoundResponse(...)`

Initial practical coverage:
- Auth:
  - `POST /auth/register`
  - `POST /auth/login`
  - `GET /auth/me`
- Users:
  - `GET /users/me`
- Categories:
  - `GET /categories`
  - `GET /categories/:slug`
- Products:
  - `GET /products`
  - `GET /products/:slug`

### 5. Add reusable OpenAPI response helpers if needed
Because the app uses global success response convention:
```json
{
  "status": "success",
  "message": "...",
  "data": {}
}
```

Add helper schemas/decorators only if needed to avoid repetitive OpenAPI response objects. Good location:
```txt
src/common/docs/api-response.schema.ts
```

Possible helper:
- `apiSuccessResponseSchema(dataSchema)`
- `apiArraySuccessResponseSchema(itemSchema)`

This keeps docs consistent without overcomplicating DTOs.

### 6. Ensure token workflow works in Scalar
OpenAPI config must include bearer auth:
```ts
.addBearerAuth(
  {
    type: 'http',
    scheme: 'bearer',
    bearerFormat: 'JWT',
    in: 'header',
  },
  'access-token',
)
```

Protected endpoints should use:
```ts
@ApiBearerAuth('access-token')
```

Expected flow in Scalar:
1. Open `/docs`
2. Call `POST /auth/login` or `POST /auth/register`
3. Copy `data.accessToken` from response
4. Paste token into Scalar auth bearer token field
5. Call protected endpoints such as `/auth/me` and `/users/me`

### 7. Add or update tests
Add e2e test for docs availability:
```txt
test/e2e/docs.e2e-spec.ts
```

Test cases:
- `GET /openapi.json` returns `200`
- OpenAPI JSON has `openapi`, `paths`, and registered endpoint paths:
  - `/auth/register`
  - `/auth/login`
  - `/auth/me`
  - `/users/me`
  - `/categories`
  - `/categories/{slug}`
  - `/products`
  - `/products/{slug}`
- OpenAPI JSON has bearer security scheme
- `GET /docs` returns `200` and HTML/text containing Scalar reference markup

### 8. Verification Commands
Run:
```bash
npm run build
npm run test:e2e -- docs.e2e-spec.ts
npm run test:e2e
```

Manual verification:
- Start app with `npm run start:dev`
- Open `http://localhost:3000/docs`
- Open `http://localhost:3000/openapi.json`
- Login/register from Scalar
- Use the access token in Scalar bearer auth
- Verify `/auth/me` and `/users/me` work with token

## Notes and Decisions
- Swagger UI remains disabled by not calling `SwaggerModule.setup()`.
- Swagger/OpenAPI is still used as the document generator, which Scalar consumes.
- Scalar UI will be the user-facing documentation.
- Existing response interceptor remains unchanged.
- Error responses can remain Nest default for now unless a later global error convention is added.
